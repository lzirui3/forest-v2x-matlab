### 问题3：你如何求解MPC优化问题？ 🟡 **MAJOR**


## 三、MPC方案的优势（公平评估）

### 优势1：理论包装更好 ✅

**为什么审稿人会更接受MPC：**
- MPC是控制论教科书的标准方法，有40年学术积累
- 有stability、robustness、constraint satisfaction的理论工具
- IEEE控制系统学会(CSS)的审稿人对MPC非常熟悉

**你可以引用的理论：**
- Terminal cost和terminal constraint保证稳定性
- Robust MPC处理不确定性
- Scenario-based MPC处理随机性

### 优势2：可以自然处理未来信息 ✅

**MPC天然适合的场景：**
- 已知未来H步的盲区地图（车辆轨迹可预测）
- 已知未来的消息到达时间（周期性CAM）
- 已知未来的中继可用性（路侧单元位置固定）

**这是启发式方法做不到的：**
启发式只看当前状态，MPC可以前瞻性规划。

### 优势3：约束处理更优雅 ✅

**硬约束vs软约束：**
- MPC可以显式处理硬约束（must满足）
- 用barrier function或惩罚项处理软约束
- 可以分析约束满足的概率保证（chance constraints）

**比启发式规则清晰：**
不再是"if-then-else"规则堆砌，而是优化问题的数学表达。

## 五、审稿人的建议：最小代价改进方案 vs 完整MPC方案

### 方案A：最小代价MPC重新表述（推荐，2-3周）⭐

**核心思路：**
将现有的Risk-constrained-v6重新表述为**贪心MPC (H=1)**

**实现：**
1. 承认现有方法是H=1的滚动时域优化
2. 目标函数保持不变，但称为"single-step MPC objective"
3. 论文中写：
   ```
   We adopt a receding-horizon optimization framework with a 
   single-step prediction horizon (H=1), where at each time k,
   the scheduler solves:
   
   a_k* = argmin_{a ∈ A} J(a | s_k)
   
   This is equivalent to greedy MPC, which is computationally 
   efficient and suitable for real-time implementation.
   ```

**优势：**
- 代码不需要大改（只是换个说法）
- 理论框架提升（从启发式→MPC）
- 可以引用MPC文献讨论性能界
- 可以说"未来工作：扩展到H>1"

**仍需补充：**
- 简单的预测模型（线性外推当前状态）
- 说明为什么H=1是合理选择（实时性要求）
- 分析H=1的性能损失（对比H=∞的oracle）

**审稿人反应：**

### 方案B：完整的多步MPC（理想，5-9周）🎯

**核心思路：**
实现真正的H≥3的MPC，证明前瞻性规划的价值。

**实现步骤：**

**Step 64.1: 预测模型设计（1周）**
```matlab
% 状态预测函数
function s_pred = predict_future_state(s_current, a_sequence, H)
    % 简单模型：线性外推 + 高斯噪声
    s_pred = zeros(H, state_dim);
    s_pred(1, :) = transition_model(s_current, a_sequence(1)) + noise;
    for h = 2:H
        s_pred(h, :) = transition_model(s_pred(h-1,:), a_sequence(h)) + noise;
    end
end
```

**预测模型选择：**
- 链路状态：一阶Markov + 距离衰落模型
- 定位质量：GDOP外推（车辆轨迹已知）
- 盲区风险：地图查询（确定性）
- 消息到达：泊松过程或确定性周期

**Step 64.2: MPC优化求解器（2-3周）**

**选项1：动态规划（精确但慢）**
```matlab
function [a_opt, J_opt] = solve_mpc_dp(s_current, H, action_set)
    % DP backwards recursion
    V = zeros(H+1, num_states);
    V(H+1, :) = terminal_cost(states);
    
    for k = H:-1:1
        for s = 1:num_states
            for a = 1:num_actions
                s_next = predict_next_state(s, a);
                cost = stage_cost(s, a) + V(k+1, s_next);
                if cost < V(k, s)
                    V(k, s) = cost;
                    policy(k, s) = a;
                end
            end
        end
    end
    a_opt = policy(1, s_current);
end
```

**选项2：采样MPC（近似但快）**
```matlab
function [a_opt, J_opt] = solve_mpc_sampling(s_current, H, action_set, K)
    % Sample K random action sequences
    best_cost = inf;
    for k = 1:K
        a_seq = random_action_sequence(H, action_set);
        [s_traj, cost] = simulate_trajectory(s_current, a_seq);
        if cost < best_cost
            best_cost = cost;
            a_opt = a_seq(1);  % Only apply first action
        end
    end
end
```

**选项3：分支定界（平衡精度和速度）**
- 用启发式函数pruning不可能的分支
- 用lower bound减少搜索空间

**Step 64.3: 实验设计（2-3周）**

**实验1：预测准确性验证**
- 在held-out数据上测试H=1,3,5步预测的RMSE
- 报告预测误差的置信区间

**实验2：时域长度消融**
- 对比H=1 (greedy) vs H=3 vs H=5
- 分析收益递减规律（H>5可能没有额外增益）

**实验3：计算时间分析**
- 报告不同求解器的平均/最坏情况延迟
- 证明满足实时性要求（<100ms）

**实验4：鲁棒性分析**
- 在预测误差±20%下测试性能
- 证明即使预测不准，MPC仍优于non-MPC

**Step 64.4: 理论分析（1-2周）**

**分析1：稳定性（如果适用）**
- 定义李雅普诺夫函数：V(s) = 当前累积风险
- 证明V(s_k) 在长期是有界的

**分析2：性能界**
- 对比H=1 vs H=∞（完整前瞻）
- 用competitive ratio量化性能损失

**分析3：约束满足概率**
- 如果用概率约束，证明满足概率≥1-ε

### 方案C：混合方案（折中，3-4周）⚖️

**核心思路：**
只在关键阶段用MPC，其他时候用现有方法。

**实现：**
```matlab
function action = hybrid_scheduler(state)
    if is_critical_phase(state)  % Emergency or PosDeg stage
        % Use H=3 MPC for lookahead
        action = solve_mpc(state, H=3);
    else
        % Use greedy H=1 for efficiency
        action = risk_constrained_v6(state);
    end
end
```

**优势：**
- 计算成本低（只在10%的时间用完整MPC）
- 仍然可以声称"MPC-enhanced"
- 可以证明在关键时刻的前瞻性价值

**实验：**
- 证明关键阶段的H=3 MPC带来主要收益
- 非关键阶段H=1足够
